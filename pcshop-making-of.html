01 緣起與痛點

從一句抱怨，變成一份需求

傳統電腦估價網頁（如原價屋、欣亞）表格密集、難在手機操作；相容性檢查仰賴使用者自身知識（買了 Intel CPU 卻配到 AMD 主機板）；估價單要手抄或截圖，無法即時分享；實體店家更常只靠紙本或 Excel 報價。

目標：設計一個「行動優先、保護式引導」的估價工具，讓新手能在兩分鐘內完成一份可以照著買的配置單，而且不會選到不相容的零件。

👤 入門新手

懂預算與用途，但不懂腳位、瓦數、尺寸；最怕買錯零件。需要系統幫他擋下錯誤。

🔧 DIY 玩家

想快速篩選、知道本地庫存、即時檢查相容性，不想被冗長表格拖慢速度。

PCShop 的品牌個性因此被定為三個詞：沈穩、精準、守護——像一位專業店員，而不是一座冰冷的資料庫。這個定位後來貫穿了配色、文案與互動行為。（來源：專案 PRODUCT.md）

02 發想與計畫

文件即設計：先寫清楚，再動手

PCShop 沒有一開始就寫程式。它先用幾份文件把「概念 → 規格 → 實作」這條鏈子釘牢，讓每一行程式都能對應到一個明確的決策。

A

PRODUCT.md — 產品概念文件

定義使用者、目的、品牌個性與設計原則：組裝進度必須隨時可見、硬衝突立即阻擋並用白話解釋、軟警告不阻擋、CPU+GPU 選定後顯示 3DMark Time Spy 預覽、漸進式揭露、拇指友善。並載明「改功能就更新文件並升版」的維護規則（目前 v1.5）。

B

規格書.txt — 技術規格書

把資料結構、標籤解析、相容性邏輯、UI 行為寫成可執行的規格（目前 v3.2），包含 WCAG AA 無障礙與 reduced-motion 支援。

C

README.md / AGENTS.md — 上手與維護守則

README 寫給開發者與店家（功能、本地預覽、後台入口）；AGENTS.md 寫給協作的 AI 代理：專案結構與維護規範。

核心原則：相容性不是寫死在程式裡的 if/else，而是一份「可配置的約束表」。營運人員改資料就能改規則，不必動程式。
03 技術棧與資料模型

刻意「無框架」：純前端 + Node 標準函式庫

為了好教、好讀、好稽核，PCShop 全程不用任何框架或打包工具。前端是 vanilla HTML/CSS/JS；後台是 Node 內建的 node:http（沒有 Express）；資料以 CSV 作為單一真實來源。

前端

Vanilla HTML / CSS / JS

無 React、無打包器。設計 token 集中在 colors_and_type.css（oklch 色彩 + 語意字級），元件樣式在 styles.css，邏輯在 script.js（約 1,200 行）。

MDN Web Docs — developer.mozilla.org
後台

Node.js（node:http，零相依）

server.js 提供靜態檔案與 REST API（/api/products CRUD），手刻 CSV 解析、cookie 工作階段，可讀寫 data/products.csv。

Node.js 官方文件 — nodejs.org
字型

Google Fonts

Inter（西文/數字）+ Noto Sans TC（繁中），透過 CDN 載入，兼顧中英文排版一致性。

Google Fonts — fonts.google.com
資料

CSV 作為真實來源

data/products.csv（59 項商品）與 data/benchmarks.csv（Time Spy 分數，以 cpu_id+gpu_id 為鍵）。瀏覽器 fetch 後在前端解析。

CSV 格式 RFC 4180

標籤驅動的規格欄位

商品的硬規格藏在分號分隔的 tags 欄位裡，載入時用正規表示式抽出腳位、記憶體類型、TDP、尺寸——後台只要編輯標籤就能改規格，前端則只收到處理後的行為旗標（badge 文字、限制規則），絕不暴露原始 tag key 給使用者。

類別關鍵欄位（由 tags 解析）
CPUSocket AM4/AM5/LGA1700、TDP / MAX TDP、記憶體支援
主機板socket、memory type、form factor（ATX / microATX / Mini-ITX / E-ATX）
顯示卡Length cm、TDP、電源接頭
機殼Max GPU Length、Max Cooler Height、支援的板型
電源wattage、模組化類型

05 溝通、調試與修正

把「相容性」做成可驗證的迴圈

調試的核心是 checkBuild()：每次加入零件都重新檢查整份配置，分出「硬衝突」（阻擋）與「軟警告」（提示不阻擋）。搭配瀏覽器 DevTools 觀察狀態、以驗收腳本回歸測試。

javascript · script.js — checkBuild()
// 硬衝突：CPU 與主機板腳位必須相符，否則阻擋加入
if (cpu && motherboard && !intersects(cpu.specs.sockets, motherboard.specs.sockets)) {
  hard.push({ categories: ["cpu", "motherboard"],
    message: `CPU 插槽 ${labelList(cpu.specs.sockets)} 與主機板 ${labelList(motherboard.specs.sockets)} 不相容。` });
}

// 硬衝突：顯示卡長度不可超過機殼上限
if (gpu && pcCase && gpu.specs.lengthCm > pcCase.specs.maxGpuLengthCm) {
  hard.push({ categories: ["gpu", "case"],
    message: `顯示卡長度 ${gpu.specs.lengthCm}cm 超過機殼上限 ${pcCase.specs.maxGpuLengthCm}cm。` });
}
調試案例 A · 軟警告閾值

電源瓦數要警告，但不該擋人

新手常買剛好夠用的電源。我們不阻擋，只在 CPU 峰值 TDP + GPU TDP + 150W 餘裕 高於電源瓦數時亮黃燈，給出明確建議瓦數。

javascript · 軟警告
const requiredWattage = (cpu.specs.peakTdp || cpu.specs.tdp || 0) + (gpu.specs.tdp || 0) + 150;
if (psu.specs.wattage && psu.specs.wattage < requiredWattage) {
  soft.push({ categories: ["psu"],
    message: `建議電源至少 ${requiredWattage}W，目前為 ${psu.specs.wattage}W。` });
}
調試案例 B · 寫 CSV 不能寫壞

後台存檔的「原子寫入」

直接覆寫——中途失敗就毀檔
// ⚠ 寫到一半若中斷，products.csv 會殘缺
await fs.writeFile(PRODUCTS_PATH, csv, "utf8");
✗ 風險：行程中斷或併發寫入會讓商品資料毀損
先寫暫存檔，再原子改名
const tempPath = `${PRODUCTS_PATH}.tmp`;
await fs.writeFile(tempPath, csv, "utf8");
await fs.rename(tempPath, PRODUCTS_PATH); // 原子操作
✓ 改名是原子操作：要嘛舊檔、要嘛新檔，永遠不會有半截檔
調試案例 C · 用驗收腳本回歸

用真實情境當測試

每次改動都跑這幾個情境（取自教學計畫的驗收章節）：

  • 給預算 25K、要能玩《英雄聯盟》＋文書，觀察新手能否獨力完成且不選到不相容組合。
  • 店家試用：在後台新增「RTX 5060」並設定庫存價格、把某品標記為「限組裝」。
  • 相容性涵蓋率：檢查十大常見衝突（如 Intel 12 代 + DDR5 主機板卻配 DDR4）是否都被擋下。
調試心法：讓錯誤「會說話」。硬衝突回傳明確訊息與建議替代品；軟警告用黃底提示而非阻擋；後台即時顯示存檔狀態。錯誤訊息本身就是最好的除錯工具。
