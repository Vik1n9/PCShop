# 電腦組裝線上估價單系統——開發方案說明書

**版本**：2.0  
**更新日期**：2026-06-05  
**設計原則**：行動優先、內部標籤驅動、結構化約束、新手無障礙

---

## 一、專案概述與目標

建置一套**以手機操作為核心**的電腦組裝估價網站，協助對硬體不熟悉或 DIY 入門的消費者，在無需具備專業知識的情況下，獨立完成零組件挑選、即時獲得相容性保障，並生成一份可購買、儲存與分享的估價單。

**核心目標**  
- 利用**內部標籤（TAG）** 與**結構化相容性規則**，實現全自動智慧篩選與衝突防堵，杜絕插槽不符、尺寸超標、瓦數不足等問題。  
- 提供直覺的**組裝進度視覺化**，清楚顯示已選／未選零件，降低使用者記憶負擔。  
- 任何零件點選後均可**立即預覽產品圖**，並可進入完整圖文介紹，無需離開估價流程。  
- 手機端操作流暢，所有元件與手勢為拇指熱區設計；桌面端提供輔助多欄視圖。  
- 後台可透過**動態標籤**靈活控制前端行為（如限組裝不單售、促銷徽章），完全不需前端程式異動。

---

## 二、目標使用者與核心設計原則

**使用者畫像**  
- 對電腦規格僅有模糊概念，害怕買錯零件的入門DIY者。  
- 完全零基礎，只想「花多少錢，得到一台能打遊戲／剪片的電腦」的一般消費者。

**設計原則**  
1. **即時回饋**：每個操作後的狀態變更（已選、衝突、建議）必須在 0.3 秒內反映於介面。  
2. **階段性資訊揭露**：預設僅顯示名稱、價格、關鍵相容規格；深入資訊透過點擊漸進展開。  
3. **防呆與引導**：硬衝突強制阻擋並提供一鍵替換；軟警告溫和提示但允許繼續。  
4. **行動優先**：所有互動以直式單手操作為基準，桌面端僅為資訊密度與排列的增強。  
5. **標籤不直接外露**：後台定義的標籤內容僅用於控制前端功能（如促銷標誌），不向使用者顯示原始標籤碼。

## 三、功能需求總覽

### 3.1 前台使用者端
- **智慧估價單主頁**  
  - 底部導航切換：組裝進度總覽、挑選零件、我的估價單記錄。  
  - 零件分為八大類：CPU、主機板、記憶體、顯示卡、儲存裝置、機殼、電源供應器、散熱器。  
- **產品清單與即時篩選**  
  - 選擇某一類零件後，系統根據已選零件自動過濾不相容品項（如主機板僅顯示相同插槽型號）。  
  - 使用者可切換「顯示全部」（包含不相容，但以灰階標示並禁點）。  
- **產品快速預覽與完整介紹**  
  - 點擊產品卡片：原地展開，顯示多張輪播圖、重點規格、價格，以及「查看完整介紹」按鈕。  
  - 完整介紹以全螢幕覆蓋層展現，內容包含行銷圖文、完整規格表、相容性提示，底部可直接「加入估價單」。  
- **視覺化組裝進度**  
  - 固定在頁面頂部的橫向進度條，清楚標示「已選 4/8」。  
  - 未選類別以圖示灰階顯示，點擊可快速跳轉至該零件挑選頁。  
- **衝突檢測與智慧建議**  
  - 加入零件時若觸發硬衝突，頂部滑出紅色警示條，說明原因並提供「一鍵篩選相容款」。  
  - 軟警告（如電源瓦數略低、無散熱器）以黃色提示列顯示，不阻擋加入。  
- **促銷與組裝限制標示**  
  - 根據內部標籤，產品卡片自動顯示「限組裝」、「熱銷」、「降價中」等徽章。  
  - 「限組裝不單售」產品無法單獨加入購物車，僅能在估價單中選配，且系統會檢查是否有對應的組裝主件（如限組裝的顯示卡需搭配整機）。  
- **估價單管理**  
  - 儲存、重新命名、分享連結、匯出 PDF／複製規格文字。  

### 3.2 後台管理端
- **商品資料管理**  
  - 每項零件除基本資訊外，需填寫**結構化相容欄位**（如 CPU 插槽、記憶體類型、最大顯卡長度等），並可附加多組內部標籤。  
  - 支援圖片上傳（主圖、輪播圖、介紹用長圖）。  
- **標籤管理**  
  - 定義標籤鍵（如 `限組裝`、`熱賣`、`即將停產`）及其類型（布林、日期、文字）。  
  - 設定標籤對應的前端行為：顯示徽章、限制單售、顯示特價倒數等。  
- **相容性約束設定**  
  - 透過視覺化下拉選單建立跨品類約束（如「CPU 插槽 = 主機板插槽」）。  
  - 立即測試：模擬選擇零件組合，列出所有約束通過／失敗狀態。  
- **估價單查詢與數據分析**  
  - 熱門零件排行、常見衝突組合，輔助進銷存決策。

## 四、資料庫與標籤系統設計

本系統採用**關鍵結構化欄位 + 輔助內部標籤**的混合模型，兼顧效能、維護性與靈活度。

### 4.1 核心商品表（products）
除名稱、售價、庫存、圖片路徑外，針對每個類別定義固定、不可為空的相容欄位（部分類別可為 NULL 但預設給值）：

- **CPU**：`socket`、`tdp`、`memory_type`（DDR4/DDR5）、`pcie_version`  
- **主機板**：`socket`、`memory_type`、`form_factor`（ATX/mATX/ITX）、`pcie_version`  
- **記憶體**：`memory_type`、`capacity`、`speed`  
- **顯示卡**：`length_mm`、`tdp`、`pcie_version`、`power_connector`（8-pin/12VHPWR）  
- **機殼**：`max_gpu_length_mm`、`mb_form_factor_support`（支援的板型列表）、`max_psu_length_mm`  
- **電源供應器**：`wattage`、`modular_type`、`length_mm`  
- **儲存／散熱器**：依情況定義必要規格（如 M.2 類型、水冷排尺寸等）。

**好處**：所有相容性篩選可直接透過簡單 SQL 條件完成，查詢速度快、資料完整。

### 4.2 內部標籤表（product_tags）
標籤完全於**後台設定**與**伺服器邏輯**中使用，前端僅根據 API 返回的「動作旗標」呈現相應 UI，**標籤原始鍵值不對使用者公開**。

- **標籤定義表 (tag_definitions)**  
  - `tag_key`（唯一識別，如 `bundle_only`、`promo`）  
  - `display_type`：`badge`（顯示徽章）、`restrict`（限制行為）、`info`（純文字提示）  
  - `config`：JSON 格式的額外參數，例如徽章顏色、文字內容、限制對象。

- **產品標籤關聯表 (product_tag_values)**  
  - `product_id`、`tag_definition_id`、`value`（若為布林可空）

**實例應用**  
| 標籤鍵 | 類型 | 作用 |
|--------|------|------|
| `bundle_only` | restrict | 該商品限整機組裝，不可單獨零售。前端「加入購物車」按鈕隱藏，估價單中若缺少其他必要零件會提示。 |
| `hot_sale` | badge | 卡片顯示「熱銷」火焰圖示，可置頂排序。 |
| `price_drop` | badge + info | 顯示「降價」標籤，並可夾帶截止日期。 |
| `clearance` | info | 提示「即將停產」，引導快速選購。 |

所有標籤邏輯在後端 API 層解析，回傳給前端的產品物件僅包含：`badge_list`（徽章文字與樣式）、`restrictions`（如 `{solo_sale: false}`），前端據此渲染，完全解耦。

### 4.3 相容性約束配置表（compatibility_constraints）
取代動態規則引擎，直接以資料列定義跨類別條件。

| 欄位 | 說明 |
|------|------|
| `source_category` | 來源零件類別 |
| `target_category` | 目標零件類別 |
| `source_field` | 來源產品表中的欄位名稱 |
| `target_field` | 目標產品表中的欄位名稱 |
| `operator` | `=`、`<=`、`IN`、`>=` |
| `is_hard` | 硬錯誤（1）或軟警告（0） |
| `message` | 給使用者的提示訊息，可用 `{value}` 佔位 |

**約束範例**  
- CPU 插槽與主機板相符：`CPU.socket = MB.socket` (硬)  
- 顯示卡長度 ≤ 機殼最大顯卡長：`GPU.length_mm <= Case.max_gpu_length_mm` (硬)  
- 電源瓦數 ≥ CPU TDP + GPU TDP + 150W：`PSU.wattage >= CPU.tdp + GPU.tdp + 150` (軟警告)  

當約束條件涉及多值（如機殼支援多種主機板板型），可將 `mb_form_factor_support` 存為 JSON 陣列，運算子使用 `IN`。

### 4.4 資料一致性保障
- 後台根據商品類別**強制顯示必填欄位**，未填寫無法上架。  
- 批次匯入時自動校驗格式，並提示缺失的關鍵欄位。  
- 約束表異動後，可一鍵重新驗證所有現有產品的相容性，產出異常報告。

## 五、智慧篩選與衝突檢測引擎

### 5.1 篩選流程（正向過濾）
1. 使用者已選零件集合 `S`。  
2. 前端請求 `GET /api/products/{category}?selected={cpu_id, mb_id,...}`  
3. 後端載入所有與 `category` 相關的約束 (`source_category` 在 `S` 中且 `target_category` = 當前類別)。  
4. 從 `S` 中擷取對應欄位值，組合成 SQL WHERE 條件，例如：  
   `WHERE socket = 'LGA1700' AND memory_type = 'DDR5'`  
5. 執行查詢，並標記不符合的商品為「不相容」（也可直接排除，由前端開關控制顯示）。  
6. 同時執行全體硬約束檢查，若有硬衝突則回傳衝突詳情。  

### 5.2 衝突回報與處理
- 當使用者嘗試將硬衝突零件加入估價單，API 回傳 `409 Conflict`，前端顯示警示橫幅並提供「篩選相容款」按鈕。  
- 若已存在之零件因後續選擇而間接衝突（如更換 CPU 導致原本相容的主機板變成不相容），系統自動將該零件標記為「待更換」並暫停估價總額計算，直到使用者解決。  
- 軟警告不阻擋，僅在頂部進度條下方顯示黃色提示列，不影響總價。

### 5.3 內部標籤驅動的行為控制
- 商品物件中附帶 `restrictions` 欄位，例如 `{bundle_only: true}`。  
- 若使用者試圖將 `bundle_only` 的顯示卡單獨加入，系統彈出提示：「此商品限組裝銷售，請先選擇主機板與 CPU」，並阻止加入。  
- 估價單儲存時檢查：若存在 `bundle_only` 零件但缺少必要主件，則顯示警告且無法提交。

## 六、用戶介面與互動設計（行動優先）

### 6.1 整體佈局（手機豎屏）
- **底部導航欄**（固定）：  
  - 🖥️ 組裝進度（總覽當前配置與缺失零件）  
  - 🔍 挑選零件（零件類別快速切換）  
  - 📋 我的估價單（歷史紀錄）  
- **頂部進度條**：常駐顯示「已選取 4／8 項」，右側為未選零件圖示清單，點擊直達。  
- **內容區域**：根據當前導航顯示對應畫面。

### 6.2 挑選零件頁面互動
- 頂部為**水平可滑動類別膠囊**（CPU、主機板…），當前選中類別高亮。  
- 產品列表採**垂直卡片流**，每張卡片顯示：  
  - 左側產品縮圖（3:2 比例）  
  - 名稱、價格、三個關鍵相容規格（如 `LGA1700`, `DDR5`, `ATX`）  
  - 右上角徽章區：由內部標籤渲染（例如 🔥熱銷、⏬降價中、🚫限組裝）  
- **點擊卡片**：原地展開，卡片高度延伸出：  
  - 圖片輪播（支援滑動，最多5張）  
  - 結構化規格摘要（以圖文形式呈現，如長度對比圖）  
  - 按鈕：「查看完整介紹」與「加入估價單」  
- 點擊「查看完整介紹」：從下至上滑入**全螢幕覆蓋層**，內部為行銷長圖、完整規格表、動態相容性備註（例如「此顯示卡可安裝於你目前的機殼」）。關閉覆蓋層後，卡片仍保持展開狀態，使用者可立即決定是否加入。  

### 6.3 組裝進度總覽頁
- 以簡化主機板線稿圖為中心，已選零件位置亮起對應顏色，未選位置顯示虛線框與類別名稱。  
- 點擊任一位置直接跳轉至該類別的挑選頁。  
- 下方列出已選零件清單，每項可長按移除或更換。  
- 明顯顯示總金額與「儲存估價單」按鈕。

### 6.4 衝突與提示呈現
- **硬衝突**：頂部進度條下方滑出紅色橫幅，附錯誤訊息與「為我挑選相容款」按鈕；衝突零件卡片邊框變紅，並在列表中以震動動效警示。  
- **軟警告**：黃色橫幅顯示建議，不影響操作。  
- **限組裝提示**：卡片徽章顯示「限組裝」；加入估價單若不合規，彈出友善說明對話框。

### 6.5 桌面端輔助佈局
- 當螢幕寬度 ≥ 1024px，介面自動切換為兩欄模式：  
  - 左欄（40%）：組裝進度總覽（固定），包含互動式爆炸圖與零件清單。  
  - 右欄（60%）：挑選零件區，產品以網格方式陳列，展開細節使用右側浮動面板，不遮擋主清單。  
- **無任何懸停效果**，所有細節展開需透過點擊。

## 七、技術架構與選型建議

### 7.1 前端
- **框架**：Next.js（React），支援 SSG/SSR，估價單分享可利用動態路由生成頁面。  
- **樣式**：Tailwind CSS，行動優先斷點設計。  
- **狀態管理**：React Query（服務端狀態） + Zustand（客戶端選取狀態、UI 互動）。  
- **動畫與手勢**：Framer Motion 處理展開收合、覆蓋層轉場；使用瀏覽器原生 Touch Events 輔助。  
- **圖片最佳化**：next/image 自動轉換 WebP，根據裝置大小調整解析度。  

### 7.2 後端
- **執行環境**：Node.js (NestJS 或 Express) 或 Python FastAPI，視團隊熟悉度。  
- **資料庫**：PostgreSQL，利用 JSONB 欄位儲存彈性規格；Redis 快取熱門篩選結果。  
- **API 設計**：RESTful，核心端點：  
  - `GET /api/products/{category}`（支援篩選參數）  
  - `POST /api/estimate/validate`  
  - `GET /api/tags/actions?product_id=`（回傳該產品對應的行為旗標）  
- **圖片儲存**：雲端物件儲存（S3 兼容）＋ CDN。

### 7.3 部署與維運
- 容器化（Docker），搭配 CI/CD 自動部署。  
- 管理後台可獨立為 SPA 或整合於同一 Next.js 專案中，透過 Middleware 進行權限控管。

## 九、總結

本方案將複雜的硬體相容性邏輯收斂至 **結構化欄位＋可配置約束表**，確保系統高效穩定且易於維護；同時將行銷與組裝限制等變動需求封裝於 **內部標籤機制**，讓營運團隊無需開發即可靈活控制前端行為。  
在用戶端，以 **行動優先的輕量互動**（展開卡片、全螢幕介紹、直覺進度條）取代繁複的懸停與跳轉，讓新手能在手機上放心地完成第一份電腦估價單，兼顧樂趣與安全感。
